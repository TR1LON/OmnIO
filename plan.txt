================================================================================
                    OmnIO — Tiered Universal Conduit System
                         Multi-Loader Minecraft Mod
                              Master Plan v1.0
================================================================================

OVERVIEW
--------
OmnIO is a multi-loader (NeoForge + Fabric) Minecraft mod providing a tiered,
universal item/fluid/energy/redstone conduit system using a bundle-block
architecture (multiple conduit types in one block). Built on the MultiLoader
Template (zero external dependencies, Java ServiceLoader for platform
abstraction). Targets Minecraft 1.21.1, Java 21.

KEY DECISIONS
-------------
- Name: "Conduit" (community-standard, universally recognized)
- Tiers: Enum-based (BASIC, ADVANCED, ELITE, ULTIMATE, CREATIVE) with
  config-overridable values (Mekanism pattern)
- Tier upgrading: Break and replace (no in-world alloy upgrading)
- Multi-loader: MultiLoader Template (common/ + fabric/ + neoforge/)
- Platform abstraction: Java ServiceLoader SPI
- Capability bridge: Thin ITransferHelper<T> SPI in common/, implemented
  per-platform (NeoForge BlockCapability / Fabric Transfer API)
- Network architecture: Persistent graph-based conduit networks
- Bundle: Multiple conduit types in a single block (up to 9)
- Per-face multi-connect: Multiple conduits of the same type can
  independently access the same face of an adjacent machine
- Polymorphic context: IConduitType.createNetworkContext() factory,
  eliminates instanceof chains in network manager
- Multi-Channel: Same-type conduits in one bundle on different color
  channels — each (type, channel) pair forms its own independent network
- Hard limit: 9 conduits per bundle block (any mix of types/channels)

================================================================================
DESIGN NOTE: MULTI-CHANNEL CONDUIT BUNDLES
================================================================================
Concept:
  A single conduit bundle block holds UP TO 9 CONDUITS (hard limit).
  Conduits of the SAME TYPE can coexist in one bundle as long as they
  are on DIFFERENT COLOR CHANNELS (16 DyeColor channels: white, orange,
  magenta, light_blue, yellow, lime, pink, gray, light_gray, cyan, purple,
  blue, brown, green, red, black).

  Each (conduitTypeId, channel) pair = one "conduit slot" in the bundle.
  Each slot forms its own independent network.

Key Innovation — Per-Face Multi-Connect:
  In most conduit mods, each face of a machine supports only ONE
  connection per conduit type. OmnIO removes that restriction.
  Multiple conduits of the same type on different channels can ALL
  independently connect to the SAME FACE of an adjacent machine.

  Example:
    Machine with 3 item inputs accessible from its back face.
    ONE bundle block behind the machine, containing 3 item conduits:
      - Red channel   → routes ores from smelter to input slot 1
      - Blue channel  → routes ingots from storage to input slot 2
      - Green channel → routes fuel from coal bin to input slot 3
    All three connect to the same face. Each has its own insert/extract
    config, priority, and filter settings.

    Diagram:
      [Smelter] ──red──┐
     [Storage] ──blue──┤ Bundle (3/9 slots) ── back ── [Machine]
    [Coal Bin] ──green─┘

Design Rules:
  - 9 conduits max per bundle (hard limit, any mix of types + channels)
  - Keying: ConduitSlot = (typeId, channel) — uniquely identifies each
    conduit within a bundle
  - Same type + same channel = same network (merge on connect)
  - Same type + different channel = isolated networks (never merge)
  - Channel = DyeColor ordinal (0-15), rendered as tint on conduit core
  - Default channel = 0 (white) on first placement
  - Channel changeable via GUI after placement

Architectural Changes Required:
  1. ConduitSlot record:
     - New record: ConduitSlot(ResourceLocation typeId, int channel)
     - Used as map key everywhere that previously used bare typeId
     - equals/hashCode on both fields

  2. OmniConduitBlockEntity:
     - Change Map<ResourceLocation, ConnectionContainer>
       to Map<ConduitSlot, ConnectionContainer>
     - Enforce size <= 9 on addConduit()
     - Per-(slot, face) connection config — multiple connections on
       the same face for different slots

  3. ConduitNetworkManager:
     - Key networks by ConduitSlot instead of ResourceLocation
     - onConduitAdded/Removed take ConduitSlot parameter
     - Neighbor matching: only merge nodes with same ConduitSlot

  4. ConduitItem:
     - Stores channel in item NBT (or component in 1.21+)
     - Default = white (0)
     - Shift+scroll to pick channel before placement (optional UX)

  5. Rendering (Phase 9):
     - One conduit core per occupied slot in the bundle
     - Core tinted by channel DyeColor
     - Connectors drawn per (slot, face) pair

  6. GUI (Phase 10):
     - Left panel: slot grid showing up to 9 conduit slots
     - Click a slot → right panel shows per-face config
     - Channel color picker (16-color palette)
     - Per-slot: insert/extract toggle, priority, redstone mode, filter

Status: DESIGNED — not yet implemented. Requires touching Phases 3, 4, 9, 10.

STEP 5: Build the Graph-Based Network Layer
================================================================================
Location: common/src/main/java/com/trilon/omnio/content/conduit/network/

  ConduitNetwork.java            — persistent graph (nodes + edges)
  ConduitNodeImpl.java           — position, connections, per-side config
  ConduitNetworkContext<T>.java  — per-network mutable state
  ConduitNetworkSavedData.java   — persistence + ticking driver

  Network lifecycle:
  ON CONDUIT PLACED:  find neighbors → join/merge networks → check block endpoints
  ON CONDUIT REMOVED: remove node → if split: BFS → split networks + redistribute
  ON CHUNK UNLOAD:    mark nodes non-ticking, keep in graph
